# 要件定義：有効リードフィルタリング機能

**プロジェクト**: lead-scraper / Salesforce  
**対象機能**: SF登録後のリード自動フィルタリング・ルーティング  
**更新日**: 2026-05-13

---

## アーキテクチャ

```
比較媒体からリード取得（Playwright）
　↓
全件SFに登録（lead-scraperは変更なし）
　↓
SFトリガー（LeadTrigger）が自動発火
　↓
LeadTriggerHandler → LeadFilterService で判定
　├─ 既存顧客   → IsExistingCustomer__c=true → SFフローでSlack通知（#CSリードチャンネル）
　├─ 重複       → 新リードをアーカイブ・既存リード備考追記
　├─ 再流入     → 新リードをアーカイブ・既存リード備考追記
　├─ いたずら   → Status=アーカイブ・first_touchpoint__c更新
　├─ 逆営業     → Status=アーカイブ・first_touchpoint__c更新（🔴 未実装）
　└─ 有効リード → 変更なし（既存SFフローでMQL通知）
```

## lead-scraperへの影響

**変更なし。** 現行通り全件SFに登録する。

---

## 判定パターン一覧（全6種）

| # | パターン | 判定条件 | 実装状況 |
|---|----------|----------|---------|
| 1 | 既存顧客 | メール or 電話で取引先責任者を照合し、紐づく契約の`churn_status__c`が空欄 | ✅ 完了 |
| 2 | 重複 | 同一メール or 電話のリードが存在し、`LeadSourceDate__c`が**半年以内** | ✅ 完了 |
| 3 | 再流入 | 同一メール or 電話のリードが存在し、`LeadSourceDate__c`が**半年超** | ✅ 完了 |
| 4 | いたずら | 電話番号が**フォーマット不正**（10桁未満 or 11桁超） | ✅ 完了 |
| 5 | 逆営業 | 問い合わせ内容に一定のキーワード・テキストを含む | 🔴 未実装（キーワードリスト待ち） |
| 6 | 有効リード | 上記1〜5のいずれにも該当しない | ✅ 完了 |

## 判定フロー

```
SFにリード登録
　↓
① 既存顧客チェック
　↓
② 重複 / 再流入チェック（半年以内 or 半年超）
　↓
③ いたずらチェック（電話番号フォーマット不正）
　↓
④ 逆営業チェック（🔴 未実装）
　↓
⑤ 有効リード → 変更なし
```

チェックは上から順に実行し、該当した時点で後続チェックはスキップする。

---

## 判定ロジック詳細

### ① 既存顧客

| 項目 | 内容 |
|------|------|
| 照合キー | メールアドレス OR 電話番号 |
| 照合対象 | SFの取引先責任者（Contact） |
| 判定条件 | 取引先責任者の取引先IDで契約（`Contract__c`）を検索し、`churn_status__c`が空欄 |
| SF処理 | `IsExistingCustomer__c = true` をセット |
| Slack通知 | SFフロー（CSリード通知）が`#CSリードチャンネル`に通知 |
| MQL通知 | ❌ なし |

---

### ② 重複 / 再流入

| 項目 | 内容 |
|------|------|
| 照合キー | メールアドレス OR 電話番号 |
| 照合対象 | SFのリード（未コンバート） |
| 重複（半年以内） | 新しいリードを`アーカイブ`に変更・既存リードの`Remarks__c`に追記 |
| 再流入（半年超） | 同上・掘り起こしとして扱う |
| `LeadSourceDate__c`がnullの場合 | 重複/再流入チェックをスキップ |
| MQL通知 | ✅ あり（既存SFフロー経由） |

---

### ③ いたずら

| 項目 | 内容 |
|------|------|
| 判定条件 | 電話番号が10桁未満 or 11桁超（数字のみ抽出して判定） |
| 備考 | SFがメールアドレスのフォーマットを自動バリデーションするため電話番号のみチェック |
| SF処理 | `Status = アーカイブ` / `first_touchpoint__c` を無効用の値に更新 |
| `first_touchpoint__c`の値 | アスピック→`アスピック（逆営業と無効）` / ミツモア→`ミツモア（逆営業と無効）` / アイミツ→`アイミツSaaS（逆営業と無効）` / その他→`逆営業と無効` |
| MQL通知 | ❌ なし |

---

### ④ 逆営業（🔴 未実装）

| 項目 | 内容 |
|------|------|
| 判定対象 | 問い合わせ内容（`web__c` / `Description`） |
| 判定方法 | **🔴 営業確認待ち**（キーワードリストの提供が必要） |
| SF処理 | `Status = アーカイブ` / `first_touchpoint__c` を逆営業用の値に更新 |
| `first_touchpoint__c`の値 | アスピック→`アスピック（逆営業と無効）` / ミツモア→`ミツモア（逆営業と無効）` / アイミツ→`アイミツSaaS（逆営業と無効）` / その他→`逆営業と無効` |
| MQL通知 | ❌ なし |

---

### ⑤ 有効リード

| 項目 | 内容 |
|------|------|
| 条件 | ①〜④のいずれにも該当しない |
| SF処理 | 変更なし |
| MQL通知 | ✅ あり（既存SFフロー経由） |

## Slack通知仕様

| パターン | 通知元 | 通知先 | 内容 |
|----------|--------|--------|------|
| 有効リード | 既存SFフロー（リードのSlack通知） | 既存チャンネル | 新規リード登録通知 |
| 既存顧客 | SFフロー（CSリード通知） | #CSリードチャンネル | CS連携通知（会社名・氏名・メール・SFリンク） |
| 重複 / 再流入 | 既存SFフロー経由 | 既存チャンネル | MQL通知 |
| いたずら | なし | - | リードイン通知のみ（既存フロー） |
| 逆営業 | なし（🔴 未実装） | - | - |

## SFオブジェクト・フィールド

### 使用カスタムフィールド（リード）

| フィールド | API参照名 | 用途 |
|-----------|----------|------|
| 既存顧客フラグ | `IsExistingCustomer__c` | 既存顧客判定結果 |
| 初回流入経路 | `first_touchpoint__c` | いたずら/逆営業時に更新 |
| 備考 | `Remarks__c` | 重複/再流入時に追記 |
| 初回流入日 | `LeadSourceDate__c` | 重複/再流入の半年判定 |

### 使用カスタムオブジェクト

| オブジェクト | API参照名 | 用途 |
|------------|----------|------|
| 契約 | `Contract__c` | 既存顧客チェック |

### 使用フィールド（契約）

| フィールド | API参照名 | 用途 |
|-----------|----------|------|
| 取引先名 | `AccountId__c` | 取引先との紐づけ |
| 解約ステータス | `churn_status__c` | 空欄=成約中と判定 |
